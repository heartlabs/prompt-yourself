# Summary Versioning — Implementation Plan

## Overview

Add a `summaryVersion` field to the `Conversation` model. When the summarizer prompt/logic changes, increment a version constant in code. The app regenerates summaries whose version is outdated (up to 30 days back), one by one, sequentially.

## Decisions

| # | Decision |
|---|---|
| 1 | `summaryVersion: Int?` = `nil` treated as version 0. First launch: set `0` for all existing summaries, **no regeneration**. |
| 2 | A dedicated `SummaryService` class with deduplication via an `inFlight` set. |
| 3 | Both `ChatViewModel` and `CalendarViewModel` call the same `SummaryService` method. |
| 4 | Go back **30 days** for version-based regeneration. |
| 5 | Show old summary while regenerating (calendar preview). Fall back to old summary if regeneration fails. |
| 6 | Sequential regeneration — one day at a time, start next only after previous completes. |
| 7 | Current version constant lives in `SummaryService` as `static let currentVersion = 0` (start at 0, bump to 1 when first prompt change happens). |

## Step-by-step

### Step 1: Add `summaryVersion` to the Conversation model

**File**: `Models.swift`

```swift
@Model
final class Conversation {
    // ... existing fields ...
    var summary: String?
    var summaryVersion: Int?  // NEW — nil = 0
}
```

SwiftData migration: automatic (new field, optional, nil by default). No data loss.

---

### Step 2: Create `SummaryService`

**File**: `SummaryService.swift` (new)

**Responsibilities**:
1. Hold a `ModelRouter` instance for making LLM calls.
2. Own the current version constant: `static let currentVersion = 0`.
3. Provide `ensureSummary(for dateKey:) -> String?` — the single entry point.
4. Provide `backfillOldVersions() async` — batch regeneration (called from ChatViewModel).
5. Maintain `inFlight: Set<String>` to deduplicate concurrent requests for the same date.

**`ensureSummary(for dateKey:)` logic**:

```
1. Fetch conversation via ConversationService
2. If conversation is nil → return nil
3. If conversation.hasRecentActivity → return existing summary (or nil)
4. Let storedVersion = conversation.summaryVersion ?? 0
5. If storedVersion >= currentVersion → return conversation.summary
6. (storedVersion < currentVersion OR summary is nil):
   a. If dateKey is in inFlight → return existing summary immediately (don't wait)
   b. Add dateKey to inFlight
   c. Generate new summary via LLM
   d. If success: save summary + set summaryVersion = currentVersion, return new summary
   e. If failure: return old summary (if any) or nil
   f. Remove dateKey from inFlight
```

**`backfillOldVersions()` logic**:

```
1. Get all past dateKeys, filter to last 30 days, most recent first
2. For each dateKey:
   a. Call ensureSummary(for: dateKey) — this checks version internally
   b. Wait 0.5s between calls
   c. If app goes to background? Task cancellation handles this naturally
```

---

### Step 3: Update `ChatViewModel`

**Changes**:

1. **Init**: Create a `SummaryService` instance (or accept one via init).
2. **`setupPersistence`**: Replace the direct `generateSummaryForPreviousDayIfNeeded()` call with `summaryService.backfillOldVersions()`.
3. **`finishReset`**: Same — call `summaryService.backfillOldVersions()`.
4. **Remove** `generateSummaryForPreviousDayIfNeeded()` — now in `SummaryService`.
5. **Remove** `generateSummary(for:service:)` — now in `SummaryService`.

---

### Step 4: Update `CalendarViewModel`

**Changes**:

1. **Add** a `summaryService` property (set during `setup`).
2. **`refreshPreview`**: The version check now goes through `summaryService.ensureSummary(for:)`:
   - If summary exists AND up-to-date → show it immediately (as before).
   - If summary exists BUT outdated → show old summary, kick off regeneration in background (fire-and-forget task). When regeneration completes, update preview state.
   - If no summary → call `ensureSummary(for:)` which generates it.
3. **Remove** local `generateSummary(for:service:)` — now in `SummaryService`.

**Version-aware preview flow**:

```
1. Load conversation
2. If today → show conversation text, return
3. If hasRecentActivity → show conversation text, return  
4. If summary exists:
   a. Show it immediately in .loaded state (old version or current, doesn't matter)
   b. If version is outdated → fire-and-forget regeneration task
5. If no summary:
   a. Set .generating state
   b. Call ensureSummary, wait for result
   c. Show new summary or fall back to conversation text
```

---

### Step 5: Update `TreeScoreService`

**No changes needed** — but note the impact:

- `gatherEntries()` reads `conv.summary` — if a summary was regenerated by `SummaryService`, it automatically picks up the new text.
- `inputSignature` changes → tree recomputes on next `loadOrCompute()`.
- **But**: the tree cache is once-per-day. So the tree won't recompute until the next day (or pull-to-refresh). This is acceptable — the improved summary will be picked up eventually.

If you want faster feedback, `TreeScoreService` could observe summary changes, but that's scope creep for v1.

---

### Step 6: Test Scenarios

| # | Scenario | Expected behavior |
|---|---|---|
| 1 | Fresh install, no data | No regeneration needed |
| 2 | Existing user, version bump 0→1 | Last 30 days regenerated sequentially, most recent first |
| 3 | User opens calendar during backfill | Calendar shows old summary, background regen replaces silently |
| 4 | User taps a date whose summary is currently being regenerated by batch | `inFlight` set → show old summary immediately |
| 5 | Regeneration fails (network down) | Old summary preserved, version not bumped, retry on next launch |
| 6 | App goes to background during backfill | Task cancelled, partial progress, next launch continues from most recent |
| 7 | User has >30 past days | Only last 30 regenerated; older days keep old summaries (shown in calendar, but LLM context uses recent 7 anyway) |
| 8 | Two consecutive version bumps (0→1→2) before all regenerations complete | Already-processed days at v1 are skipped (v1 < v2), still regenerated — okay, idempotent cost |
